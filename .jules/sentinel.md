## 2024-05-24 - [PostgreSQL RLS Privilege Escalation]
**Vulnerability:** The Row Level Security (RLS) policies for the `users` table allowed any user to insert or update their own profile without restrictions on the `role` or `suspended` columns.
**Learning:** This allowed users to grant themselves `admin` role or un-suspend themselves by simply providing those values when inserting or updating their own profile via Supabase API. RLS only controls *which* rows a user can access or modify, but the `WITH CHECK` clause is required to restrict *what* data they can write into those rows.
**Prevention:** Always restrict security-sensitive columns (like role, permissions, status) when writing RLS `INSERT` and `UPDATE` policies by adding appropriate assertions in the `WITH CHECK` clause to ensure they match default safe values or that only privileged users can modify them.
## 2024-06-13 - [PostgreSQL RLS INSERT Mass Assignment]
**Vulnerability:** The RLS INSERT policies for `families`, `events`, and `join_requests` only checked authentication status, allowing users to inject arbitrary values for security-sensitive columns (e.g., creating a pre-approved family or faking the `created_by` field).
**Learning:** Checking `auth_uid_text()` in the `USING` or `WITH CHECK` block is not enough if you don't map that identity to the corresponding column in the row being inserted. RLS policies control *which* rows can be written, but the expressions inside the `WITH CHECK` must constrain *what* data is written to those rows.
**Prevention:** Always explicitly bind `auth_uid_text()` to ownership columns and enforce default safe values for status columns in the `WITH CHECK` constraints for `INSERT` operations. Note: Be careful when applying this to `UPDATE` operations to avoid recursive table queries.
