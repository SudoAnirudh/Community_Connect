## 2024-05-24 - [PostgreSQL RLS Privilege Escalation]
**Vulnerability:** The Row Level Security (RLS) policies for the `users` table allowed any user to insert or update their own profile without restrictions on the `role` or `suspended` columns.
**Learning:** This allowed users to grant themselves `admin` role or un-suspend themselves by simply providing those values when inserting or updating their own profile via Supabase API. RLS only controls *which* rows a user can access or modify, but the `WITH CHECK` clause is required to restrict *what* data they can write into those rows.
**Prevention:** Always restrict security-sensitive columns (like role, permissions, status) when writing RLS `INSERT` and `UPDATE` policies by adding appropriate assertions in the `WITH CHECK` clause to ensure they match default safe values or that only privileged users can modify them.
## 2024-06-12 - [PostgreSQL RLS Mass Assignment & Privilege Escalation Bypass]
**Vulnerability:** The Row Level Security (RLS) `INSERT` and `UPDATE` policies across multiple tables (e.g., `families`, `join_requests`, `events`, `reports`) lacked `WITH CHECK` clauses. This allowed malicious authenticated users to manipulate fields they shouldn't control, such as self-approving their `join_requests` (setting `status` to 'approved'), or creating a family and assigning someone else as the admin (`admin_uid`).
**Learning:** In Supabase/PostgreSQL, RLS policies that only check if a user is authenticated (`public.auth_uid_text() is not null`) allow *any* authenticated user to insert *any* data unless strictly bounded. A missing `WITH CHECK` clause leaves the database vulnerable to mass assignment and privilege escalation. Also, when self-referencing a table in a subquery within a `WITH CHECK` policy (like checking the old state of a row), the table must be aliased to avoid scoping issues that result in multiple rows being returned.
**Prevention:** Always enforce safe defaults and strict ownership assertions within the `WITH CHECK` block for `INSERT` and `UPDATE` RLS policies (e.g. `status = 'pending'`, `created_by = public.auth_uid_text()`). Ensure subqueries in RLS policies correctly alias the inner tables.
